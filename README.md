# Product Funnel & Retention Command Center for E-commerce

## 1. Название проекта

Product Funnel & Retention Command Center for E-commerce — портфолио-проект по продуктовой аналитике интернет-магазина косметики на событийных данных.

## 2. Бизнес-вопрос

Как пользователи проходят путь от просмотра товара до покупки, какие сегменты дают наибольший вклад в выручку и какие точки роста можно найти через воронку, удержание и пользовательские признаки?

## 3. Данные

Источник данных: Kaggle dataset `eCommerce Events History in Cosmetics Shop`.

Сырые CSV-файлы нужно положить вручную в `data/raw/`. Автоматическая загрузка данных из Kaggle в проекте не используется.

Ожидаемый тип данных: события пользователей интернет-магазина, включая просмотры, добавления в корзину, покупки, товары, категории, пользователей, сессии или признаки, достаточные для их восстановления.

## 4. План анализа

1. Проверить структуру и качество сырых событий.
2. Подготовить очищенный слой данных в `data/processed/`.
3. Выполнить sessionization и восстановить пользовательские сессии.
4. Построить SQL-витрины для продуктовых метрик.
5. Рассчитать воронку `view -> cart -> purchase`.
6. Построить cohort retention по первым покупкам или первым активностям.
7. Сегментировать пользователей по поведению и ценности.
8. Подготовить baseline purchase propensity model.
9. Собрать BI-dashboard и бизнес-рекомендации.

## 5. KPI

- Количество пользователей.
- Количество сессий.
- Доля сессий с просмотром товара.
- Конверсия `view -> cart`.
- Конверсия `cart -> purchase`.
- Конверсия `view -> purchase`.
- Средний чек.
- Выручка.
- Retention по когортам.
- Доля повторных покупателей.
- Purchase propensity score для пользовательских сегментов.


6. Запустить проверки:

```powershell

```

## Подготовка данных

Для ETL нужны CSV-файлы из Kaggle dataset `eCommerce Events History in Cosmetics Shop`. Положите в `data/raw/` месячные файлы событий, например:

- `2019-Oct.csv`
- `2019-Nov.csv`
- `2019-Dec.csv`
- `2020-Jan.csv`
- `2020-Feb.csv`

ETL читает все `.csv` из `data/raw/`, объединяет события, приводит `event_time` к datetime, приводит `price` к float, удаляет полные дубли, исключает строки с отрицательной ценой и сохраняет результат:

```text
data/processed/events_clean.parquet
```

Запуск:

```powershell
python scripts/run_etl.py
```

## SQL-витрины

SQL-слой строится в DuckDB на основе файла `data/processed/events_clean.parquet`.

Запуск:

```powershell
python scripts/build_marts.py
```

Скрипт создает базу:

```text
data/marts/ecommerce.duckdb
```

И экспортирует витрины в `data/marts/` в форматах parquet и CSV.

Схема витрин:

- `fct_events` — очищенные события с датой события и исходными продуктовыми полями.
- `fct_sessions` — пользовательские сессии с количеством событий, просмотров, добавлений в корзину, покупок, выручкой, длительностью и числом уникальных товаров и категорий.
- `mart_funnel_daily` — дневная воронка по категории и бренду: просмотры, корзины, покупки, конверсии и выручка.
- `mart_retention` — cohort retention по месяцу первой покупки.
- `mart_product_conversion` — конверсия и выручка по товарам.

## Аналитика

Основной аналитический ноутбук:

```text
notebooks/01_product_analysis.ipynb
```

Запуск:

```powershell
jupyter notebook notebooks/01_product_analysis.ipynb
```

Ноутбук читает данные из `data/marts/ecommerce.duckdb` и использует готовые витрины из DuckDB. Перед запуском аналитики нужно выполнить:

```powershell
python scripts/run_etl.py
python scripts/build_marts.py
```

Финальные продуктовые выводы должны опираться только на рассчитанные метрики. Если значение еще не рассчитано, используется шаблон значения: `[X]%`, `[Y] п.п.`, `[Z] рублей`.

## 8. Что будет в dashboard

- Обзор ключевых метрик: пользователи, сессии, покупки, выручка, средний чек.
- Воронка `view -> cart -> purchase` с конверсиями по этапам.
- Cohort retention по периодам.
- Сегменты пользователей по активности, покупкам и ценности.
- Динамика заказов и выручки.
- Товары и категории с наибольшим вкладом в продажи.
- Бизнес-рекомендации по росту конверсии и удержания.

## 9. Ограничения анализа

- CSV-файлы не входят в репозиторий и добавляются вручную.
- Выводы зависят от полноты и качества исходных событий.
- Метрики без расчетов в коде не должны подставляться вручную; для них используются шаблоны значений: `[X]%`, `[Y] п.п.`, `[Z] рублей`.
- Результаты baseline-модели используются как аналитический ориентир, а не как production-решение.
